# Barka — AI-Native Progressive CMS — Agent Rules

## Mission — context for all decisions

Barka is the #1 CMS for marketing-driven companies that need rich, multi-language websites across multiple domains and publish a lot of content fast. It was born from frustration with Payload, Astro, Drupal, WordPress, and Strapi — where AI agents struggled with multi-language, multi-site, custom landing pages, and high-volume publishing. Every decision should optimize for: AI agents creating content and features easily, multi-language/multi-domain as first-class, rapid landing page and blog post creation, and zero friction from idea to published page.

## INVARIANTS — NEVER violate

- NEVER delete or overwrite content/, config/, themes/ — USER-OWNED directories
- NEVER modify starters/ without syncing changes to ALL starters (lokatech + kadoservices + blank)
- NEVER hardcode language prefixes (/pl/, /de/) in theme components — use _url() and _t()
- NEVER commit apps/smoke-crm/ — temporary test fixture
- ALWAYS run `npm test && npm run typecheck` before push
- ALWAYS verify with Playwright before reporting a fix as done
- ALWAYS check INVARIANTS.json before modifying protected paths

## Architecture decisions (settled — do not re-litigate)

- Theme resolution order: active theme → base theme → starter → built-in fallback
- ALL section components use _url() for href links, _t() for UI text (hero, cta, features, blog-listing, text-with-image, etc.)
- init --force cleans content/, config/, themes/ before copying
- barka build must mirror app.ts rendering pipeline exactly
- Zero JS on frontend — public pages ship 0 bytes of JavaScript
- UI translations via config/translations/`<lang>`.yaml — never hardcode per-language text in JSX
- Multi-site: config/sites.yaml defines domains, content filtered by siteId + langcode
- Starters are self-contained distributions — after init, user owns everything

## Content structure

- Markdown files with YAML frontmatter in content/
- Landing pages use YAML sections (content/landing-pages/*.yaml)
- PL content: file suffix (.pl.md, .pl.yaml) — URLs get /pl/ prefix from framework, NOT from content

## Theme components (SDC)

- Each component: directory with template.tsx + schema.yaml
- Components receive: { data, settings, themeSettings }
- Use `themeSettings._url?.(path)` for localized links
- Use `themeSettings._t?.(key)` for translated UI strings
- Component CSS in colocated .css files, served via /static/components.css

## npm package model

- Package name: `@barkajs/barka`
- content/, config/, themes/ are gitignored — generated by `barka init`, user-owned
- starters/ and src/built-in-theme/ ship with the npm package
- CLI resolves starters from cwd first, then from package root (getPackageRoot())

## Commands

```
barka dev                 # Dev server with hot reload
barka build               # Static HTML to dist/
barka init -s X           # Init from starter
barka db:init             # SQLite + admin user
barka create <name>       # Scaffold new project
barka starters            # List available starters
npm test                  # Vitest (must pass before push)
npm run typecheck         # TypeScript (must pass before push)
```

## Commit format

`type(scope): description` — e.g. `feat(sections): add gallery section type`
